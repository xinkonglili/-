go里面的vendor

1、先下载govendor工具：go get -u github.com/kardianos/govendor

2、工程目录要放在gopath/src下面

3、执行govendor init，自动生成vendor文件夹(存放项目需要的依赖包)和vendor.json文件

4、如果vendor.json中有的包不存在，需要从gopath文件夹添加到vendor目录下，执行 govendor add +external

5、如果有些包还是没有，可以执行govendor add 某个包

//有无模块支持，项目中使用vendor的时候，GO111MODULE应该设置成off

  • GO111MODULE=off无模块支持，go会从 GOPATH 文件夹和 vendor 目录中寻找依赖项。
  • GO111MODULE=on模块支持，go忽略 ​GOPATH 文件夹，只根据go.mod 下载依赖。
  • GO111MODULE=auto在 $GOPATH/src 外层且根目录有go.mod 文件时，开启模块支持；否者无模块支持。

在win11的窗口使用：set GO111MODULE=off，修改为off

然后运行bee run即可运行

如果想让自己的代码不暴红，就go mod init创建go.Mod文件

定义：守护进程是在后台运行的特殊的进程，通常用于长期运行的服务和任务，他们会自动和终端分离，默默在后台执行，不和用户进行交互

https://www.cnblogs.com/jumpspider/p/15912700.html

如果 daemon 参数为 true，则代码将以守护进程的方式运行。守护进程是在后台运行的进程，通常与系统服务或长时间运行的任务相关。守护进程在后台运行，并且通常不与控制台交互。

如果 forever 参数为 true，则代码将永久运行，不会退出。这表示代码将持续运行而不受限制，直到显式地被终止或出现无法恢复的错误

为什么创建子进程并让父进程退出，子进程就可以成为一个独立的守护进程呢

os.Getppid() == 1,说明这个进程的父进程是init进程，那么这个进程就是守护进程

创建子进程并让父进程退出是守护进程的一种常见实现方式。通过这种方式，子进程会成为一个独立的守护进程的原因如下：

  1. 继承了父进程的会话和进程组：当父进程创建子进程时，子进程会继承父进程的会话和进程组。父进程退出后，子进程成为孤儿进程，并且它的父进程会被设置为 init 进程。这使得子进程成为一个独立的进程，不再与父进程或终端会话相关联。
  2. 脱离了终端会话：父进程退出后，子进程不再与终端会话相关联。这意味着它不会受到终端关闭或用户注销的影响。守护进程通常需要在后台长时间运行，而不受终端会话的影响。
  3. 重定向了标准输入、标准输出和标准错误：在创建子进程时，可以将子进程的标准输入、标准输出和标准错误重定向到适当的文件描述符或文件中。这样可以确保守护进程的输入输出不依赖于终端，而是可以与其他文件或设备进行交互。

通过创建子进程并让父进程退出，可以使子进程在后台独立运行，脱离终端会话，并具备守护进程的特性。这种方式常用于守护进程的实现，以确保守护进程的稳定运行和独立性。

需要注意的是，这只是守护进程的一种常见实现方式，具体的实现方式可能因操作系统和需求而有所不同。在实际开发中，可以根据具体要求选择适合的方式来创建守护进程。

这个函数可以用于创建一个子进程并执行指定的命令。通过将命令的路径和参数传递给 SubProcess 函数，您可以创建一个与父进程相互独立的子进程来执行该命令。子进程的标准输入、标准输出和标准错误将被重定向到与父进程相同的标准输入、标准输出和标准错误。

这个函数的目的可能是为了封装创建子进程并执行命令的过程，以便在其他代码中可以方便地调用。

需要注意的是，该函数的实现假设传入的参数中的第一个元素是要执行的命令的路径，后续的元素是命令的参数。

在给子进程设置标准输入、标准输出和标准错误时，将它们重定向到父进程的标准输入、标准输出和标准错误的代码，更适用于那些需要与父进程进行交互或共享输入输出的情况，而不是典型的守护进程场景。

对于守护进程，通常会将标准输入、标准输出和标准错误重定向到其他地方，例如将其重定向到日志文件中，或者将其关闭以避免不必要的输出。这样可以使守护进程在后台默默地执行任务，而不与终端或其他进程进行交互。
