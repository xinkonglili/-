为什么创建子进程并让父进程退出，子进程就可以成为一个独立的守护进程呢

os.Getppid() == 1,说明这个进程的父进程是init进程，那么这个进程就是守护进程

创建子进程并让父进程退出是守护进程的一种常见实现方式。通过这种方式，子进程会成为一个独立的守护进程的原因如下：

  1. 继承了父进程的会话和进程组：当父进程创建子进程时，子进程会继承父进程的会话和进程组。父进程退出后，子进程成为孤儿进程，并且它的父进程会被设置为 init 进程。这使得子进程成为一个独立的进程，不再与父进程或终端会话相关联。
  2. 脱离了终端会话：父进程退出后，子进程不再与终端会话相关联。这意味着它不会受到终端关闭或用户注销的影响。守护进程通常需要在后台长时间运行，而不受终端会话的影响。
  3. 重定向了标准输入、标准输出和标准错误：在创建子进程时，可以将子进程的标准输入、标准输出和标准错误重定向到适当的文件描述符或文件中。这样可以确保守护进程的输入输出不依赖于终端，而是可以与其他文件或设备进行交互。

通过创建子进程并让父进程退出，可以使子进程在后台独立运行，脱离终端会话，并具备守护进程的特性。这种方式常用于守护进程的实现，以确保守护进程的稳定运行和独立性。

需要注意的是，这只是守护进程的一种常见实现方式，具体的实现方式可能因操作系统和需求而有所不同。在实际开发中，可以根据具体要求选择适合的方式来创建守护进程。

要将最新版本的IPFS与IPFS 0.8.0兼容并在同一集群中实现数据共享，您需要进行以下步骤：

  1. 升级IPFS：首先，您需要将IPFS 0.8.0升级到最新版本。请从IPFS官方网站（https://ipfs.io）下载并安装最新版本的IPFS。

  2. 更新配置文件：在升级完成后，您需要更新IPFS节点的配置文件以确保兼容性。请打开IPFS节点的配置文件（通常位于~/.ipfs/config）并进行以下更改：

    • 将配置文件中的API和Gateway端口更改为不与现有IPFS节点冲突的端口。确保每个节点具有唯一的端口配置。
    • 确保Swarm部分的Addresses列表中包含新版本IPFS节点的地址，以便节点可以连接到集群。

  3. 启动新节点：使用更新后的配置文件启动新版本的IPFS节点。确保每个节点都在不同的机器上运行，并且具有唯一的节点ID。

  4. 加入集群：在新版本的IPFS节点上运行以下命令，使其加入到IPFS 0.8.0节点所在的集群中：

    ipfs-cluster-ctl peers add /ip4/<旧节点IP地址>/tcp/<旧节点Swarm端口>/ipfs/<旧节点ID>
```

将`<旧节点IP地址>`替换为IPFS 0.8.0节点的IP地址，`<旧节点Swarm端口>`替换为旧节点的Swarm端口，`<旧节点ID>`替换为旧节点的节点ID。

  5. 数据共享：一旦新节点成功加入到集群中，您就可以使用IPFS的数据共享功能了。将文件添加到其中一个节点上，它将自动在整个集群中进行复制和分发。

异步，同步，并发，并行

  • 同步：需要等到前一个任务执行完毕之后，后面的任务才能进行（同步中并没有并发或者并行的概念）
  • 异步：不同的任务之间并不会相互等待，是可以同时进行的，实现异步的方式–多线程，如果启动多个线程，计算机是多核的，就可以分配到独立的核心上运行，实现真正的并行，如果使用单核的处理器，或者通过亲和力强制将线程绑到某个核心cpu上，通过分配时间片的方式，这种线程是并发的执行
  • 并发：计算机可以同时执行多个任务，怎么做到并发有不同的形式，单核cpu可以通过给不同的任务分配时间片的方式，让一个任务执行一段时间，切换到另外一个任务，不同的任务会交替往复的执行下去，这个过程也叫做进程或者线程的上下文切换
  • 并行：多个任务同时执行，不需要通过分配时间片的方式，多核处理器，可以实现真正的并行

Map是不是并发安全的，如何实现并发安全的

不是并发安全的，sync.map是安全的，使用读写分离实现的，其思想是空间换时间

RWMutex在某一时刻由任意数量的reader持有，或者是只被单个的writer持有,当有一个goroutine持有读锁，此时有其他的goroutine想要持有写锁，这个goroutine会被阻塞，直到读锁被释放

sync.Map的四个方法：Store(存值)、Load(取值)、Delete、Range

Load：func (m *Map) Load(key interface{}) (value interface{}, ok bool)

Load方法用来加载sync.Map中的值，入参是key，返回值是对应的value以及value存在与否

sync.Map更适合读多，更新多，而插入新值少的场景

readOnly 不适合反复插入，读取新值的场景，因为这种场景会频繁操作dirty，需要频繁加锁和更新read

简介

Sync包下的map很像是Go的map[interface{}]interface{}，但是对于普通的map来说，sync.Map可以保证在多个goroutine的并发使用是安全的，无需额外添加锁来进行控制

方法

func( * Map)Store(key,value any)

将一个key,value存入map中

func(* Map)Delete(key any)

将一个key从map中删除

func(* Map)Load(key any) (value any, ok bool)

从map中获取一个Key

func(* Map)LoadAndDelete(key any) (value any, loaded bool)

从map中获取一个key，并且将key删除，如果key存在那么loaded会返回true，并且value就是map中存的值，如果key不存在那么loaded会返回一个false，并且value为nil

func (m *Map) LoadOrStore(key, value any) (actual any, loaded bool)

从map中加载一个key，如果key不存在，那么就会将value存放在key中，返回value，loaded为false，如果key存在则返回value，loaded为true

func (m *Map) Range(f func(key, value any) bool)

迭代出map中的key，value,如果回调函数中返回false，那么就会停止迭代，如果返回true，那么就会继续迭代

控制子协程的几种方式：

  • Channel: 使用channel控制子协程 make(chan int)
  • WaitGroup : 使用信号量机制控制子协程
  • Context: 使用上下文控制子协程

Context作用

1、跨goroutine传递请求范围的值，这样可以避免使用全局变量或传递大量参数来实现数据共享

2、取消信号传递：通过Context，可以向相关的goroutine传递信号(取消信号，中止信号)，这样可以避免因为某些操作超时或出现错误导致程序出现不必要的等待或者阻塞

3、截止时间管理：Context还可以设置超时时间，控制某些goroutine的执行时间，执行超过时间，Context会自动发出取消的信号

小tips：struct{}{}不占用内存空间，0大小的数据结构，在某些情况下可以用来当作占位符或者信号量

a、信号量：某些情况我们可能只会关心某些事情的发生，但是不需要传递实际的数据
